常用命令总结

  • hexo new "文章标题": 创建新文章。
  • hexo clean: 清理文件。
  • hexo g (generate): 生成静态网页。
  • hexo s (server): 启动本地预览。
  • hexo d (deploy): 部署网站。
  • hexo clean && hexo g && hexo d: 清理、生成、部署。

Hexo 部署基本流程

使用 Hexo 部署博客到 GitHub Pages 的标准命令流程通常是：

  1. 清理 (Clean): 清理 Hexo 生成的缓存文件和旧的静态文件。

  2. 生成 (Generate): 根据 Markdown 源文件和主题，生成最新的静态网页文件到 public 目录。

  3. 部署 (Deploy): 将 public 目录中的静态文件推送到远程仓库（如 GitHub 仓库）的指定分支。

通常会将这三个命令连起来执行：hexo clean && hexo g && hexo d。

常见问题：fatal: Authentication failed

在执行 hexo d 命令进行部署时，可能会遇到 fatal: Authentication failed 的错误。这通常是由于 Git 在连接 GitHub 仓库时认证失败引起的。主要原因和解决方法如下：

  1. 原因： 直接使用 GitHub 账号密码进行 HTTPS 认证（GitHub 已不再支持）。 解决方法：
    • 使用 Personal Access Token (PAT): 前往 GitHub 账号设置生成一个具有仓库访问权限的 PAT。在 Git 提示输入密码时，输入 PAT 而不是账号密码。
    • 切换到 SSH 认证： 配置 SSH Key，将公钥添加到 GitHub 账号，并在 Hexo 的 _config.yml 中将 repo 地址改为 SSH 格式 (git@github.com:你的用户名/你的仓库名.git)。

确定博客访问网址

成功部署到 GitHub Pages 后，博客的访问网址取决于你的 GitHub 仓库类型：

  1. 用户或组织网站：
    • 仓库名称： 必须是 你的用户名.github.io 或 你的组织名.github.io。
    • 部署分支： 通常是 main 或 master 分支。
    • 访问网址： https://你的用户名.github.io 或 https://你的组织名.github.io。
  2. 项目网站：
    • 仓库名称： 任意其他名称（例如 myblog）。
    • 部署分支： 通常部署到 gh-pages 分支（Hexo 的 Git 部署插件通常会自动处理）。
    • 访问网址： https://你的用户名.github.io/你的仓库名 (例如 https://wangruozhou.github.io/myblog)。

重要提示： 确保你的 Hexo 站点配置文件 _config.yml 中的 url 设置为你实际的博客访问网址，以便 Hexo 正确生成网站内部链接。

通过理解这些步骤和常见问题，你应该能更顺利地使用 Hexo 部署你的个人博客。
